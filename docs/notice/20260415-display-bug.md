# 【已修复】按次计费版显示异常说明

> 发布日期：2026-04-15 &nbsp;|&nbsp; 影响范围：按次计费版用户 &nbsp;|&nbsp; 状态：**已修复**

## 问题描述

官方仪表盘当前存在次数显示异常：进度条的已用值（C）被放大了约 **4 倍**，总量值（M）在接口层面也从 500 变为 2000（500 × 4），但页面仍显示为 500。

因此您可能看到进度条已满（如 500 / 500），但**实际额度并未用完**。

## 实际情况

管理员后台已统一查看，各账号的**实际扣费次数正常**。显示异常不影响使用。

## 截图对比

下面以同一账号为例，展示仪表盘与接口的数据差异。

### 仪表盘页面（数据异常）

仪表盘显示 500 / 500（已满），但实际接口返回 `used: 584, limit: 2000`，真实进度应为 584 / 2000。

![仪表盘显示异常 — 进度条已满但实际未用完](/images/usage-dashboard.png)

### /api/usage 接口（数据正确）

同一账号访问 `/api/usage`，显示 `numRequests: 146, maxRequestUsage: 500`，真实已用仅 146 次。

![API 接口返回的真实用量数据](/images/usage-api.png)

## 如何确认真实用量

在已登录账号的浏览器中访问：

```
https://cursor.com/api/usage
```

关键字段：

- **numRequests** = 实际扣费次数
- **maxRequestUsage** = 总可用次数

更详细的说明请参阅 [如何查看真实用量](/guide/check-usage)。

## 换算参考

- 粗略换算：**仪表盘显示值 ÷ 4 ≈ 真实值**
- 部分账号不是严格 4 倍，因为单条请求可能计费 1 次或 2 次
- 参考标准：**思考模型 = 2 次，非思考模型 = 1 次**

## 官方回复

官方社区支持工程师已确认此问题为 **Bug**，由近期代码变更导致，不影响实际套餐限额和收费方式，后续将修复。

> 您看到的数字 "2000" 实际上是以美分表示的金额（20.00 美元），被错误地显示为请求次数。您实际的请求限制（500）没有改变，限制执行不受影响。此事已上报团队，目前正在内部跟踪调查。

![官方社区支持工程师回复截图](/images/official-reply-display-bug.png)
