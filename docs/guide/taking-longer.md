# Taking longer than expected

对话过程中出现 **"Taking longer than expected"** 提示，通常是因为编辑器到 Cursor 服务器的网络连接不稳定。

## 原因分析

Cursor 的对话采用**流式输出**，模型的回复是逐字实时推送到编辑器的。如果您的网络节点与 Cursor 服务器之间存在**丢包、超时或断流**，就会导致推送中断——相当于 Cursor 正在回复，但您收不到内容。

一旦连接中断，对话就会卡住并显示该提示，需要手动点击「继续」才能恢复。

![用户反馈：对话卡住后需要手动点继续](/images/chat-group-network.png)

## 解决方法

### 第一步：检测网络状况

Cursor 内置了网络诊断工具，**不消耗次数**，可以放心使用。

**操作路径**：设置 → **Network** → **Run Diagnostic**

检测结果会显示到 Cursor 服务器的连接延迟和稳定性。如果出现错误或延迟过高，说明当前网络节点不适合使用 Cursor。

### 第二步：切换网络节点

- 切换到一个与 Cursor 服务器连接稳定的节点后，**重新执行网络检测**确认
- 建议避开中国香港、中国台湾节点
- 建议开启**全局代理 + 增强模式（TUN 模式）**，确保编辑器的所有流量都经过代理

### 第三步：启用 HTTP/2（进阶优化）

开启 HTTP/2 可以有效缓解流式输出中断的问题，但需要您的代理工具支持 HTTP/2 并且正确配置。

::: tip 提示
大部分用户的代理工具默认不支持 HTTP/2 或未正确配置。如果您不确定，可以先尝试前两步，通常已经能解决问题。
:::

## 总结

| 排查顺序 | 操作 | 说明 |
| --- | --- | --- |
| 1 | 网络诊断 | 设置 → Network → Run Diagnostic（不耗次数） |
| 2 | 切换节点 | 避开港台，选择连接稳定的节点 |
| 3 | 全局 + TUN | 确保编辑器流量走代理 |
| 4 | 开启 HTTP/2 | 进阶优化，需代理工具支持 |
